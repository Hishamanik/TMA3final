Introduction

The final assignment of COMP466 required me to develop 4 web applications. The first of these web applications is a frontend application that would use cookies to track how many times a user visited the application, in addition to displaying their IP address and timezone.

The second web application is similiar to assignment 1. This application would run a slideshow, using a list of pictures stored on the server. It includes buttons to start and stop the slideshow, and change the transition from a sequential ordering to a random one.

The third and fourth parts revolve around building a computer store. This store contains a multitude of features. Users are able to browse existing models, look up specific parts (such as graphics and Cpus) and then select and customize that computer. The customization allows a user to change any number of components to either more expensive or different ones. The price is then updated dynamically as a user modifies their system.

Finally, a user can add a computer to a cart, which can be removed. In part 3, the design required me to not use a database, but rather hardcoded values. Part 4 extended this basic system by adding a database, user authentication, and orders.

Analysis and design of the web application: For part 3, I attempted to model the UX flow of Amazon. On Amazon, there exists a list of products that you can select for more information. Once a user has selected a product, they can sometimes customize it (change color, size etc). Finally, users can add items to their cart, and then are able to view their cart. Items in their cart can be deleted, and the price updates dynamically. Since part 3 required me to hardcode data rather than use a DB, I kept the data within a JSON file that I read via a set of functions in api/api.py. This was the only section of my codebase interacting with data, making the switch to a DB in part 4 much quicker and simplier.

Research: The assignment required to look at some online stores and analyze the layout. I went to Amazon.ca and noticed that there is a search bar and filter type of catagory menu with different category options. The home page contains categories and display items and a list of categories along with welcome messages and naviagtion bar to naviate to all other options in the website. When hovering over one of the left menu items, it then display subcategories of the hovered category each with other subcategories. This design shows the intelligence down by the developer which allows the user to their desired product with the least clicks possible to remove confusion about where to find it and having to switch from page to page to find the desired products. When I ignore the subcategories and click the components category, it loads a new page but display all if the expandable submenu items and their categories from the previous page as a vertical menu on the left side of the page. On the component page, the search bar and menu at the top of the page remain there like on the first page.The bar with the search and logo and sign in/registration and cart buttons is fixed to the top of the screen for constant access. So to summarize, the layout of Amazon.ca is consistent accross each page with easy follow and easy access which people with very minimal intelligence can use also.

My Interpretation of the assignment project and requirements: Finally, a user can add a computer to a cart, which can also be removed. In part 3 design I used rather hardcoded values as stated in the Assignment description but in Part 4 I had to extend this basic system by adding a database, user authentication, and orders.

The specification table contains references to components, which is an abstract model (non-existent in SQL). Instead, there is an individual table for each component, which is referenced by the computer spec.

In this part, I first changed up api.py to use Django's QuerySet API to interact with the models. After replacing the old code interacting with the JSON file, I added functionality for creating a computer which is assigned to a user. This would allow me to populate a user's cart and order.

For authentication, I used Django's build in system, which uses the User model. This required me to develop login/signup screens. Once a user logs in, they can access all screens in part 4. If a user is not logged in, requests to any path other than /login or /create are redirected to the login page.
